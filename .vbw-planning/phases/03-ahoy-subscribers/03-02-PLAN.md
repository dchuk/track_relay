## Notes

**Why Wave 2 (sequential) instead of parallel with 03-01:** The 0.3.0 release commit is the contract that REQ-09 is fully satisfied. If the client subscriber lands and bumps to 0.3.0 before the server subscriber is green, the release is broken in half. Sequential dependency ensures the version bump only happens after both halves are working.

**Why direct edit of `version.rb` instead of `scripts/bump-version.sh`:** Project CLAUDE.md explicitly forbids running `scripts/bump-version.sh` without explicit user request. Direct edit is the safe path; the script can be run by the user when they decide to publish.

**`init()` API breaking change rationale:** AhoyJs has no equivalent of GA4's `measurementId` â€” its dispatch target is `window.ahoy.track` which auto-initializes from the host app's `<script src="ahoy.js">`. Forcing AhoyJs-only hosts to pass a fake measurementId would be worse UX than the breaking change. Documented under `### Changed (BREAKING)` in CHANGELOG so SemVer-conscious hosts get a clear migration signal at the 0.3.0 boundary (which is pre-1.0, so breaking changes within minor versions are within the SemVer-0.x convention).

**Cross-subscriber name parity:** Server `Subscribers::Ahoy.name` returns `"TrackRelay::Subscribers::Ahoy"`; client `AhoyJs.name` is `"AhoyJs"`. The names differ but the dispatched event names are byte-identical (server `payload.name.to_s` â†’ `tracker.track`, client `eventName` â†’ `window.ahoy.track`). REQ-09's "same event names as the server" criterion is about the event-name string, not the subscriber name â€” both halves pass the catalog Symbol through as-is.

## Notes

**Why `tsup` for the dual-build (Fix 4 â€” honoring 02-CONTEXT line 52):** The 02-CONTEXT discussion committed to dual ESM+CJS. The previous draft would have shipped a `package.json` `exports` map declaring CJS but only an ESM source â€” `require('@track_relay/client')` would crash on consumers. `tsup` is the lightest tool that solves this: zero config beyond `entry`/`format`/`outDir`, no TypeScript required (it ingests plain `.js`), and produces both `dist/index.mjs` (ESM) and `dist/index.cjs` (real CJS with `module.exports`) in a single command. Alternatives considered: raw `esbuild` (more wiring), `swc` (faster but two passes for ESM+CJS), `rollup` (overkill). `tsup` adds ~1 dev dependency and produces audit-clean output. Tests run against `client/src/` (the source) for fast feedback; `build_smoke.test.js` is the single place that asserts `dist/` artifacts exist after `npm run build`. This keeps the test loop fast (no rebuild on every test edit) while still catching build regressions in CI.

**Why init requires `measurementId` (Fix 3 â€” JSâ†”Rails source-of-truth):** The Rails layer owns the GA4 property identifier (`config.ga4_measurement_id`). The JS package CANNOT read Ruby config. Three viable wiring patterns exist: (a) a `<meta name="track-relay-measurement-id" content="...">` tag, (b) inline `<script type="module">init({measurementId: "<%= ... %>"})</script>`, (c) a global `window.TRACK_RELAY_CONFIG`. Option (b) was chosen as the documented default because it's the same pattern as `manifestUrl` (already passed via `asset_path()` in the layout), keeps state out of `window` globals, and gives the host app a single ERB block to maintain. The required-both contract on `init()` makes misconfiguration loud â€” passing only one throws synchronously rather than silently dispatching events that GA4 can't route.

- **Plan 02-03** for the manifest schema (the JS validator must match the typed schema shape produced by `Manifest.generate`),
- **Plan 02-04** for the version bump being meaningful â€” releasing `0.2.0` requires the GA4 server subscriber to actually exist.
File-disjoint with Plan 02-04 (which lives in Wave 3): this plan only touches `client/**`, `.github/workflows/ci.yml`, `lib/track_relay/version.rb`, `CHANGELOG.md`, `README.md`. Plan 02-04 touches `lib/track_relay/subscribers/ga4_measurement_protocol.rb`, `delivery_job.rb`, `subscribers/base.rb`, `configuration.rb`, `errors.rb`, `lib/track_relay.rb`, `linter.rb`, `tasks/track_relay.rake`. **No file overlap â€” could in theory run in Wave 3, but version bump must be the LAST commit of the phase, so sequencing wins.**

The `client/` directory layout (vs `js/` or `packages/client/`) is chosen because: (a) it's flat at repo root which is conventional for small companion packages (e.g. `actiontext/app/javascript/` style); (b) Scout Â§7 line 703 used `packages/client/` but a single-package monorepo is unnecessary overhead for v0.2.0; (c) future packages (`@track_relay/server-debug`, etc.) can move to `packages/` in Phase 4 if needed.

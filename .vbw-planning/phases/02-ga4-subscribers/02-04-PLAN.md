## Goal

Ship the server-side `Ga4MeasurementProtocol` subscriber that POSTs validated events to GA4's Measurement Protocol endpoint, with proper retry policy on transient failures and call-time payload constraint enforcement (REQ-27 split). Wire it through the existing `DeliveryJob` for async delivery with `retry_on` / `discard_on` mapping to typed exceptions. Implements REQ-08 + REQ-10 (call-time half) + REQ-11.

## Notes

**Why `ga4_delivery_attempts` is NOT configurable in 0.2.0 (resolved Fix 2 contradiction):** The original draft tried to have it both ways — declare a config attr AND use a literal `5` because the config singleton might not exist at class-load time. That's a real load-ordering hazard: `DeliveryJob` is `require`d before user `track_relay.rb` initializers run, so `TrackRelay.config.ga4_delivery_attempts` would be unsafe at the `class < ActiveJob::Base` body level. Cleanest fix for 0.2.0: ship `TrackRelay::DeliveryJob::DEFAULT_GA4_DELIVERY_ATTEMPTS = 5` as a class-local constant on `DeliveryJob` and DROP the config attr entirely. A future minor (Phase 4) can introduce runtime configurability via an `after_initialize` hook or by moving `retry_on` to a `self.inherited`-style late-binding wiring. Documented as a deferred idea — not a regression.

**Why `Subscribers::Base` carve-out is required (Fix 1 — narrow REQ-23 exception):** Phase 1's REQ-23 mandates that `safe_deliver` swallows ALL `StandardError`s when `swallow_subscriber_errors = true`, returning the exception object as a value. `DeliveryRetriableError` and `DeliveryDiscardableError` inherit from `StandardError`, so without an explicit re-raise they get swallowed. ActiveJob's `retry_on`/`discard_on` only fire when an exception PROPAGATES out of `#perform` — and `DeliveryJob#perform` calls `safe_deliver`, sees a returned-exception value (not a raised one), and would NOT trigger retry. The fix is whitelist-style: rescue these two classes FIRST and re-raise, then fall through to the existing rescue. This is a single 2-line change in `subscribers/base.rb`. Without it, the entire retry policy is silently broken in production (the only environment where `swallow_subscriber_errors` defaults to true).

This plan intentionally does NOT modify `track_relay.gemspec` to add an HTTP client dependency — `Net::HTTP` is in Ruby's stdlib and is sufficient for a single low-latency POST. If profiling later shows we need connection pooling, swap to `net-http-persistent` in Phase 4. Wave-3 isolation is required because:
1. Touches `delivery_job.rb`, which Plan 02-02's `controller_tracking.rb` change does not, but the GA4 subscriber's behavior depends on `Current.client_id` being populated by the resolver chain (Plan 02-02).
2. Touches `configuration.rb` again (Plan 02-02 already added `client_id_resolvers`); sequential modification across waves prevents merge conflict.
3. Touches `subscribers/base.rb` for the retry-passthrough carve-out; Plan 02-01 (Wave 1) also touches `subscribers/base.rb` for `only:`/`except:` filter wiring. Sequential across Wave 1 → Wave 3 — no merge conflict.
4. The `lib/track_relay.rb` require addition is at a different position than Plan 02-02's `client_id/*` requires — append-only at the subscriber-block region.
